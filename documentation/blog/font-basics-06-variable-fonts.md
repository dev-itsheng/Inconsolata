# 可变字体把变化装进一个文件

现代字体文件不只保存“每个字长什么样”。它还可以保存一个连续设计空间，让字重、宽度、倾斜或光学尺寸在同一套字体里变化。

可变字体最适合放在点阵和轮廓之后理解。只有先知道 glyph 是轮廓、metrics 和字体表的组合，才容易理解：所谓可变，不是把一堆 TTF 文件压进一个文件，而是让同一套字形在设计轴上插值。

## 传统字体家族靠多个文件分工

传统字体家族常常按文件分发：Regular 一个文件，Bold 一个文件，Condensed 一个文件，Italic 又是另一个文件。用户在软件里看到的字体家族，其实可能由很多静态实例组成。

这种方式很直观，也很稳定。设计师可以单独调整每个字重，每个文件都可以独立测试。但它也有成本：文件多，实例数量有限，Web 加载重，应用想做连续动画或响应式排版时不够灵活。

静态字体还有一个体验问题：用户只能在设计者预先导出的实例里选择。设计者提供 Regular、Medium、Bold，用户就只能选这些；网页想在窄屏上略微收窄字宽，或者在深色背景里微调字重，也只能换到已有文件。可变字体把这些离散实例之间的空间也开放出来。

## 可变字体把变化放进设计轴

Microsoft OpenType 规范的 [Font Variations Overview](https://learn.microsoft.com/en-us/typography/opentype/spec/otvaroverview)把可变字体描述为沿设计轴连续变化的字体。常见轴包括：

- `wght`：字重，从细到粗。
- `wdth`：宽度，从窄到宽。
- `slnt`：倾斜。
- `opsz`：光学尺寸，不同字号使用不同细节。

可变字体的关键不是“自动加粗”。真正的可变字体需要设计师准备 master，让同一个 glyph 在不同位置都有合理形态。插值只能连接已经设计好的空间，不能替代审美判断。

在网页里，这类轴可以通过 CSS 暴露出来，例如：

```css
.code-sample {
  font-family: "Inconsolata", monospace;
  font-variation-settings: "wght" 650, "wdth" 92;
}
```

这段代码表达的是“取同一套可变字体，在字重 650、宽度 92 的位置生成实例”。它不是给 Regular 套一个粗体滤镜，也不是把字体横向压扁；真正的结果取决于字体文件里有没有对应轴和可靠的变化数据。

为了能插值，不同 master 之间通常要保持兼容结构。简单说，同一个 glyph 在不同 master 里需要有可对应的轮廓点和轮廓顺序。否则从 Light 走到 Bold 时，曲线可能扭曲，部件可能穿插，复杂字可能出现奇怪变形。可变字体看起来像一个文件，背后仍然是严谨的设计空间。

这也解释了为什么可变字体不是压缩包。它不是把 Regular、Bold、Condensed 这些文件塞在一起，而是在字体表里保存默认轮廓、变化数据、轴信息和实例信息。渲染器根据用户选择的轴位置计算当前 glyph 应该长什么样。

## 光学尺寸说明字体不是简单缩放

`opsz` 这条轴尤其适合说明字体的复杂性。小字号和大字号不是同一个轮廓简单缩放。

小字号需要更大的 x-height、更开阔的 counter、更明确的笔画间隙和更稳的像素表现。大字号可以保留更细腻的曲线、更紧的间距和更精致的细节。传统金属活字时代，不同字号常常有不同字模；可变字体把这种思想重新放进数字字体。

中文里也有类似问题。小字号正文需要内部空间更清楚，复杂字的笔画要避免粘连；大字号标题可以保留更细腻的笔形、起收笔和结构张力。很多读者以为字体缩放只是几何变换，但真正好的字体会考虑不同尺寸下人眼看到的黑度、间距和细节。

## Inconsolata v3 本身就是可变字体项目

Inconsolata v3 的源码配置里有 `wdth` 和 `wght` 两条轴。Ligconsolata Next 继承这条线，所以后续新增 glyph 时也要注意：一个 glyph 不能只在 Regular 看起来正常，还要考虑不同 master、不同字重和不同宽度下是否合理。

这对编程连字尤其重要。`=>`、`<==`、`====` 这类 glyph 可能由横线、箭头端点、斜线、竖线和空白组成。如果 Regular 看起来对，Bold 里横线太黑，Condensed 里端点太挤，仍然不能算完成。

等宽可变字体还要守住列宽。`wght` 轴可以改变笔画粗细，`wdth` 轴可以改变整体宽度，但同一个实例里每个字符仍然要遵守等宽逻辑。编程连字更麻烦：它既要在不同轴位置看起来合理，又要保持原始字符序列的总 advance width。这个约束会直接影响生成脚本和 review 清单。

## 可变字体对中文和代码字体都有长期价值

中文字体字符量大，做完整字重家族成本很高。可变字体能减少分发文件数量，也让系统、网页和设计软件更灵活。但中文可变字体并不容易：每个字都有复杂结构，插值时要保证部件不碰撞、重心不漂移、笔画不糊。

代码字体也有类似问题。程序员会在不同屏幕、字号、主题和缩放比例下使用字体。字重、宽度、斜体和符号清晰度都影响阅读。可变字体让代码字体有机会更细地适配用户偏好，但前提是每个变化都经过真实场景验证。

JetBrains Mono 这类现代代码字体提供多字重和斜体，说明开发者阅读场景已经不只需要一个 Regular。浅色主题、深色主题、Retina 屏、低分辨率外接屏、终端、IDE、投影演示，对字重和清晰度的要求都不同。可变字体可以让这些选择更连续，但项目仍然需要 specimen、字号矩阵、字重矩阵和真实编辑器验证。
